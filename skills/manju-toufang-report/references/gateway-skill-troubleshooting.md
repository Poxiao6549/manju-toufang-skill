# Skill "不触发" 的排查思路

微信/Telegram 等常驻平台（gateway）上，用户提问却像没加载 skill（模型反问"数据在哪"、当普通问题处理）时，按以下思路排查。

## 排查顺序：先排除"模型行为"，再看"注入层"

### 第一步：判断是注入问题还是模型行为问题

最常见的"不触发"原因其实不是 skill 没加载，而是**模型加载了 skill 但卡在某个业务误判上**没往下走。

**排查方法**：
- 检查模型是否调用了 skill_view/skills_list 工具
- 检查模型的 reasoning（如果可见）卡在哪一步
- 常见卡点：模型误判了某个前置条件（如认证状态、租户选择），反问用户而不是直接执行

**如果是模型行为问题** → 改 SKILL.md 的指令消除误判（例如明确"忽略某字段，直接执行"），不需要排查注入层。

### 第二步：确认 skill 是否被注入到系统提示

- 确认 skill 在 skills 列表中显示为 enabled
- 确认当前平台/会话的工具集（toolset）包含 skill 相关工具（skill_view 等）
- 如果工具集不含 skill 工具，模型看不到任何 skill 清单

### 第三步：常驻进程不热加载 skill

常驻进程（gateway）在启动时加载一次 skill 索引，**不热更新**。新建/修改 skill 后需要重启常驻进程才能生效。

**注意事项**：
- 必须把所有 skill 改动全部完成后再重启，否则可能读到旧缓存
- 重启方法参考所用平台的官方文档

## 本 skill 踩到的具体案例：currentTenantId 假阴性

模型加载了 adex-ks skill，其中有"未设默认租户就停下来让用户选租户"的规则。模型用 `adex user` 返回的 `currentTenantId: 0` 误判为"没设默认租户"，于是卡住反问用户，报告链路中断。

**真相**：`currentTenantId`（服务端账户字段，恒为0）≠ 本地默认租户（配置文件里已设定，命令实际用的）。本地租户设好后，不传 `--tenant` 也能正常拉到数据。修复 = 在 SKILL.md 明确"忽略 currentTenantId:0，租户已锁定，直接拉数不要反问"。

**通用教训**：当"数据源认证/配置状态"有两个字段（一个服务端、一个本地），别让 skill 拿服务端只读字段当"是否已配置"的判据 —— 用能真正驱动命令的那个（本地 config / 实际查询是否成功）。
